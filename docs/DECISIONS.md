# DECISIONS — ADR (Architecture Decision Records)

Формат: Контекст → Решение → Последствия/альтернативы. Отклонения от тех-спеки фиксируются здесь (правило §0 «гипотезы, не аксиомы»).

---

## ADR-0001 — Хостинг: self-host docker-compose на Hetzner exit-fi вместо Vercel
**Дата:** 2026-07-01
**Контекст:** Тех-спека дефолтом предлагает Vercel + Supabase Cloud. У владельца есть свой сервер exit-fi (89.167.127.0, Hetzner EU, 2 vCPU / 3.7 GB), где уже работает внутренняя VPN-нода. Задача — стартовать дёшево и с полным контролем, migration-ready.
**Решение:** Приложение = изолированный `docker-compose` стек на exit-fi: Caddy(:443) → Next.js(`next start`) → postgres+pgvector. Сборка локально, на сервер уезжает образ. Отдельная docker-сеть `remlab-net`, щедрые лимиты памяти, swap 4 GB — чтобы не задеть VPN-ноду.
**Последствия:** Дёшево, полный контроль, переезд = копирование compose+env+volume-dump+образ. Минусы: сами держим uptime/бэкапы/масштаб; сосед — боевой VPN (изоляция обязательна). Альтернатива (Vercel+Supabase) остаётся fallback.

## ADR-0002 — БД: самохостед postgres:17 + pgvector в контейнере (не Supabase Cloud)
**Дата:** 2026-07-01
**Контекст:** Спека завязана на Supabase (Auth/Realtime/Storage/pgvector). Полный self-host Supabase (~8 контейнеров, ~4 GB) не влезает рядом с VPN-нодой на 3.7 GB.
**Решение:** Держим только `postgres:17 + pgvector` в контейнере (volume + ночной pg_dump). Auth/Realtime/Storage реализуем по мере фич: интерим — anonymous session id + polling статуса job (спека допускает polling как fallback Realtime). Полноценный Supabase-стек либо отдельные контейнеры (GoTrue и т.п.) — позже, при необходимости, отдельным ADR.
**Последствия:** Лёгкий футпринт, полная автономия БД. Минус: часть удобств Supabase (готовый Auth/Realtime/RLS-тулинг) реализуем руками. Drizzle + Zod + RLS в самом Postgres — сохраняем.

## ADR-0003 — TLS: Let's Encrypt через TLS-ALPN-01 на :443 (Caddy), без открытия :80
**Дата:** 2026-07-01
**Контекст:** Домен remont-lab.online — A-запись GoDaddy напрямую на сервер (без Cloudflare-прокси). На сервере iptables INPUT=DROP; :443 открыт, :80 закрыт снаружи.
**Решение:** Caddy выпускает и авто-продлевает сертификат Let's Encrypt через TLS-ALPN-01 (challenge на :443). :80 не открываем.
**Последствия:** Автопродление из коробки, минимум изменений firewall. Fallback: открыть :80 для HTTP-01, если ALPN будет капризничать.

## ADR-0004 — Лимиты памяти контейнеров: щедрые (страховка blast-radius)
**Дата:** 2026-07-01
**Контекст:** exit-fi несёт боевую VPN-ноду. Без лимитов OOM-killer при аварии remlab может убить rw-core/remnanode и уронить VPN.
**Решение:** mem_limit: remlab-app 1 GB, postgres 1 GB (shared_buffers=128MB), caddy 128 MB. Высокие настолько, что в норме не достигаются — ловят только runaway. Подтверждено владельцем.
**Последствия:** VPN-нода защищена (OOM бьёт по нашему контейнеру, он рестартует). Ничего не «режется» в нормальной работе.

## ADR-0005 — Дисковая гигиена / автоочистка
**Дата:** 2026-07-01
**Контекст:** Источники переполнения: логи контейнеров, старые образы после деплоев, рост БД.
**Решение:** (1) ротация логов Docker json-file max-size=10m max-file=3; (2) очистка старых образов после деплоя (последние 2 тега remlab-app + prune dangling, scoped — VPN-образ не трогаем); (3) weekly systemd-timer remlab-cleanup (dangling + build-cache + stopped remlab); (4) ротация pg_dump-бэкапов (последние 7); (5) df-watchdog при >80%; (6) swappiness=10.
**Последствия:** Диск не переполняется сам, VPN-образы не затрагиваются (без глобального `system prune -a`).

## ADR-0006 — Целевая платформа образов: linux/arm64 (сервер aarch64)
**Дата:** 2026-07-01
**Контекст:** При первом деплое обнаружено, что exit-fi — **ARM (aarch64)** (Hetzner ARM/Ampere), а не amd64. Образ, собранный на amd64-машине разработчика, не запускался (app в Restarting; «platform linux/amd64 does not match host linux/arm64»).
**Решение:** Все образы remlab собираем под `linux/arm64` кросс-сборкой: `docker buildx build --platform linux/arm64 --load` + эмуляция `tonistiigi/binfmt --install arm64` + builder `remlabx`. `deploy.sh` настраивает и делает это сам.
**Последствия:** Сборка на amd64 идёт под QEMU-эмуляцией (медленнее — минуты, приемлемо). Альтернатива на будущее — self-hosted arm64 GitHub Actions runner или нативная сборка на самом сервере (но там мало RAM — держим сборку локально). Migration-ready сохраняется: на другой arm64-хост образ переносится как есть; на amd64-хост — сменить `--platform` в `deploy.sh`.

## ADR-0013 — Трейсинг AI-пайплайна (подробный лог + «номер генерации»)
**Дата:** 2026-07-02
**Контекст:** Впереди сотни тестов пайплайна распознавания. Модели будут меняться (Nano Banana → ChatGPT → ControlNet/SD), промпты переписываться регулярно (свой промпт под сценарий), сам пайплайн — разным под сценарии. Нужно: (1) подробный лог каждого вызова LLM в разрезе пути (вход/выход/настройки/модель на каждом шаге, включая промежуточные фото); (2) человекочитаемый «номер генерации» на экране; (3) по номеру — краткий+подробный разбор (промпт, путь, модели, исходное и итоговое фото); (4) команда/скилл «генерация N»; (5) на MVP — кнопка жалобы пользователя со всем логом.
**Решение:**
- **Захват — в слое провайдеров.** `getImageProvider/getVisionProvider` оборачиваются инструментированно (`lib/providers/traced.ts`): каждый вызов LLM пишет шаг в активный прогон. Прогон открывает оркестратор через `runWithTrace` (`lib/trace/recorder.ts`), ambient-контекст — AsyncLocalStorage (`lib/trace/context.ts`). Нет прогона → passthrough. Смена модели/промпта/шага НЕ требует правок системы лога — новый вызов логирует себя сам.
- **Данные:** `generation_runs` (seq = номер генерации через sequence `generation_seq`, pipelineId+version, статус, итоги), `generation_steps` (stepName, kind, provider, model, promptId+version+текст, params, input/output-текст, latency, cost, ошибка), `generation_assets` (картинки input/intermediate/output — **байты на диске/томе**, в БД только ссылка; base64 в БД не пишем).
- **Версионирование:** реестр промптов `lib/prompts/registry.ts` (id+version+build) и реестр пайплайнов `lib/pipelines/registry.ts` (сценарий = шаги+модель+промпт, дефолт `preview-v1`) — точка расширения под новые модели/сценарии.
- **Сжатие:** сервис **imagor** (Go+libvips, internal-only на remlab-net, как в SUP2 ADR D18) уменьшает фото (`fit-in 1536`, webp q80) ПЕРЕД LLM → меньше пикселей = меньше токенов; сохранённый input-ассет = то, что реально ушло в модель. `IMAGOR_BASE_URL` пуст → шлём исходник (best-effort).
- **Разбор:** скилл `/trace` (триггер «генерация N»), `pnpm trace <N>` (`tools/trace-show.mjs`), admin-роуты `GET /api/trace/<N>` + `/api/trace/asset/<id>` за `TRACE_ADMIN_TOKEN`; «Генерация #N» на `/preview`; кнопка «Сообщить о проблеме» (`POST /api/trace/report`). Ретеншн `TRACE_RETENTION_DAYS=90` (`pnpm trace:prune`).
- **Дисциплина:** path-scoped правило `.claude/rules/pipeline-tracing.md` — трейсинг обязан оставаться полным при смене модели/промпта/шага (бампать версии реестров).
**Альтернативы (отклонены):** внешний LLM-observability (Langfuse и т.п.) — лишняя зависимость и данные вне периметра; ручное логирование в каждом модуле — рассинхрон при эволюции пайплайна; base64-картинки в БД — раздувание БД (диск сервера мал).
**Решение владельца:** хранить фото на диске; сделать все фазы сразу; ретеншн 90 дней; «номер генерации» — простая сквозная нумерация.
**Последствия:** логирование best-effort (ошибка записи не валит пайплайн); секреты в лог не пишем. Ops при деплое: `pnpm db:migrate` (trace-таблицы+sequence), сервис imagor + том `remlab-traces`, env, `trace:prune` на таймер. imagor-образ под arm64 проверить. Фото — ПДн (TODO legal, не блокируемся).

## ADR-0014 — Пивот бизнес-модели v0.2 → v0.3 (affiliate-first freemium)
**Дата:** 2026-07-02
**Контекст:** Владелец передал Master Project Brief v0.3 (`docs/master-brief-v0.3.md`) — мастер-документ с приоритетом выше документов v0.2. Меняются монетизация, freemium-граница и приоритеты.
**Решение:**
- **Freemium-граница переопределена:** бесплатно — полноценный подбор **до N (3–5) реальных товаров из фидов с открытыми реферальными ссылками** (двигатель affiliate, доход ~3%, сеть Гдеслон) + 1 preview + товарный бюджет; платно — **«комната целиком»** (все элементы 8–20 + 3 альтернативы) + сервисный слой (Renovation Cost Engine со стоимостью работ, чек-лист, план, PDF, workspace). Товары и ссылки БОЛЬШЕ НЕ прячутся за paywall.
- **Три ступени монетизации (складываются):** (1) affiliate (бесплатный продукт); (2) paid room pack — розница ~2 990 ₽ (тест 1 990/2 990/4 990, якорь 1/4–1/3 цены дизайнера) и дизайнеры-B2B ~990 ₽/комната от 10 (batch, выгрузки, white-label позже); (3) застройщики «квартира+ремонт в одной ипотеке», ~5% с чека ремонта (~150 тыс. ₽/клиент, vision/Stage 4).
- **Product matching:** «генерация → поиск максимально похожего в большой базе фидов» по embeddings (pgvector); промпты ограничены каталого-представимыми объектами; метрика similarity (порог) с первого дня как предиктор refund/доверия. Не inpainting в Stage 1.
- **Рынок РФ (фиды Гдеслон) → UK** (Awin/CJ), архитектура locale-agnostic (валюта/фиды/сети/rates через абстракции).
- **Канал:** дешёвый информационный трафик DIY-кластера (~10 ₽ CPC) + SEO (SSR/SSG, программатик). Главная falsifiable-гипотеза юнит-экономики: 1 атрибутированная покупка на ~45 визитов при чеке ~30 000 ₽ → доход ≈ 2× CAC; проверка в Stage 0 с kill-критерием (CAC > 2× дохода → пересборка).
- **Roadmap v0.3:** Stage 0 (concierge/валидация) → Stage 1 (Free Matching MVP) → Stage 1B (Paid Room Pack) → Stage 1.5 (Designer Wholesale) → Stage 2 (материалы+расширение) → Stage 3 (подрядчики) → Stage 4 (застройщики).
**Уточнения владельца (2026-07-02) — граница free/paid = уровень «что сделать с комнатой» (3 варианта):**
1. **«Освежить без ремонта» — БЕСПЛАТНО:** визуализация + подбор до 3 реальных предметов мебели по визуализации (открыто, реф-ссылки, можно заменить). Без материалов/сметы/плана.
2. **«Недорого обновить» — ПЛАТНО ~1 490 ₽** (розница; владелец называл ~1000 — параметр): мебель в любом количестве + подбор материалов. БЕЗ расчёта стоимости ремонта, без чертежей/дизайнера.
3. **«Ремонт под ключ» — ПЛАТНО 9 900 ₽** (гипотеза; рынок дизайн-проекта в РФ сверен 2026-07-02 ~45–100 тыс ₽/комната Москва, ~25–50 тыс регионы, «под ключ» с ремонтом от 150 тыс+): мебель + материалы + гайды + информация для строителей (чертежи) + смета (Renovation Cost Engine) + услуга живого дизайнера. По сути замена работы дизайнера, но в 5–10× дешевле.
- Ссылки на товары — реферальные во всех вариантах (affiliate сохраняется). Материалы подняты в платные Stage 1-варианты (в брифе §5 были Stage 2).
- **Код-долг:** структура «3 открыто, остальное закрыто» (`modules/ideas`, `FREE_VISIBLE=3`) верна для варианта 1; нужно заменить seed-заглушки на реальные товары из фидов Гдеслон + реф-ссылки, добавить платные варианты 2/3 (материалы, любое кол-во, чертежи, Cost Engine, услуга дизайнера) и согласовать уровни «что сделать» с `interventionLevel`. Трактовка «раздать все товары бесплатно» ошибочна.
- **Postgres self-host подтверждён владельцем** — «Supabase» из брифа НЕ берём (ADR-0001/0002); трактуем как «Postgres+pgvector». Конфликт закрыт.
**Альтернативы (отклонены):** сохранить v0.2-границу «товары за paywall» (душит affiliate); inpainting реальных товаров в Stage 1 (сложно/дорого).
**Последствия:** новые воркстримы Stage 1 — пайплайн фидов Гдеслон (загрузка→нормализация→embeddings→ресинк), affiliate-трекинг (click_id→постбэк), метрика similarity, лимиты/anti-abuse генераций, второй вход «похожая мебель по фото», SEO. Открытые параметры к тестам: N (3/5), цена room pack, порог similarity, лимит генераций. Блокеры до платного трафика: условия Гдеслон (атрибуция web→app МП, «оформлен» vs «выкуплен», реальные ставки).
