# Memory Automation — как история проекта попадает в Memory Bank

> Наработки по ведению памяти проекта. Задача: чтобы ВСЁ проектное durable оседало в `.memory_bank/`
> (в git, видно команде), а не в служебной локальной памяти Клода, откуда оно теряется при `/clear`.

## Правило №1 — единственное хранилище
`.memory_bank/` — единственный источник истории проекта. Служебный каталог harness
`~/.claude/projects/<sanitized-cwd>/memory/` хранилищем НЕ считается: он вне репо, команда его не видит.
Полное правило — `.claude/rules/memory-discipline.md` (auto-loaded).

## Цикл сессии (что гарантирует полноту)
1. **Старт** — читать `INDEX.md` → нужные доки.
2. **В процессе** — durable-факт фиксировать сразу в `.memory_bank/` + no-orphan в `INDEX.md`.
3. **Конец сессии** — `/memory-consolidate`: свести проектное из диалога и (если есть) из локальной
   памяти harness в `.memory_bank/`; затем `/memory-check`.

## Скиллы (инструменты)
- `/memory-init` — первичная сборка Memory Bank из `_intake/` (bootstrap проекта с нуля).
- `/memory-consolidate` — свод сессии в `.memory_bank/` (замена авто-хуку; запускает Клод — без рекурсии/runaway).
- `/memory-check` — быстрый структурный гейт: regen decision tree, orphan/stale/битые ссылки (Node не нужен).
- `/memory-cleanup` — глубокая уборка: дубли/устаревшее/архивация (dry-run → подтверждение).

## Почему не авто-хук на каждый SessionEnd
Авто-хук записи контента опасен (рекурсия, стоимость, нужно суждение о том, что durable). Поэтому свод —
через скилл, который запускает Клод в конце сессии (см. memory-discipline «Цикл сессии»). Консистентность
проверяет `/memory-check` / `tools/memory-audit.mjs`. Опциональный Stop-хук (только проверка, не запись)
описан в `INIT.md` kit-а и требует Node.

## Старт разработки «с нуля» в этом проекте
Memory Bank самодостаточен: новая сессия читает `CLAUDE.md` → `INDEX.md` → `project-state.md` и может
продолжать/начинать разработку. Скиллы `/memory-*` установлены в `.claude/skills/`.
