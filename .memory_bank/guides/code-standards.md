# Стандарты кода — полная версия

Тонкая версия — `.claude/rules/code-standards.md`. Дефолт стека — **TypeScript + Zod** (под соло-
вайбкодинг). Не-TS проект — адаптируй (`/memory-init`).

## Must-haves
- TypeScript `strict: true`; без `any`/неявных. Публичные интерфейсы — типизированы явно.
- Named exports (если конвенция стека это поддерживает) — лучше для поиска/рефакторинга.
- Сетевые вызовы изолированы в слое доступа к данным (`services/`/`lib/`); UI их не делает.
- Один источник формы данных: Zod-схема → тип через `z.infer` (не дублировать DTO руками).
- Простые понятные типы; без сверхсложных generic без необходимости.

## Zod на границах доверия (по умолчанию)
Валидируем КАЖДЫЙ внешний вход схемой Zod — это и DTO, и рантайм-защита (типы TS на рантайме не
проверяются). Границы: route handlers / API, server actions, формы, парсинг файлов/CSV/Excel, env.

- `const dto = schema.safeParse(input)` (или `.parse`) на входе → дальше только провалидированный `dto`.
- Тип — `z.infer<typeof schema>`. Схемы ПЛОСКИЕ и явные, минимум магии.
- **Разделение ответственности:** Zod валидирует ФОРМУ/типы; БИЗНЕС-правила (обязательно-если-
  применимо, контрольные суммы, уникальность) — в отдельном слое валидации/доменке, НЕ в схеме.
- Object-схема по умолчанию СТРИПАЕТ неизвестные ключи на `.parse()` → бесплатный mass-assign guard
  (server-controlled поля просто не включай в схему).
- Группируй обвязку в хелперы (`intField`/`numField`/`textField`), но саму схему держи явной
  (видно каждое поле и его тип).
- ⚠️ **Гоча (стоила бы порчи данных):** `z.preprocess` выполняется и на ОТСУТСТВУЮЩЕМ ключе
  (`undefined == null` → true) → для частичного патча превращает absent в `null` и СТИРАЕТ поле.
  Хелперы делай без `preprocess` (через `z.union([z.null(), z.literal("").transform(()=>null), …])`).
  ОБЯЗАТЕЛЬНЫЙ тест на схемы автосейва: частичный патч `{одно_поле}` → в выходе ТОЛЬКО это поле.

## tRPC и подобное — по необходимости, не по умолчанию
Если фуллстек на server actions (напр. Next App Router) — типобезопасность вызовов УЖЕ есть; tRPC
добавляет generic-слой + react-query без выгоды. Вводить только при отдельном клиенте/публичном API.

## Структура файла/модуля
1. Импорты. 2. Типы. 3. Основная сущность (компонент/функция/класс). 4. Экспорт.
Связанное — рядом; несвязанное — в отдельные файлы.

## Именование
| Сущность | Конвенция | Пример |
|----------|-----------|--------|
| Компоненты/классы | PascalCase | `UserCard` |
| Функции/переменные/хуки | camelCase | `useAuth` |
| Константы | SCREAMING_SNAKE | `MAX_SIZE` |
| Файлы-сервисы/модули | по конвенции стека | `authService.ts` |

## Запреты
- Дублирование >10 строк → вынести в общую функцию/модуль.
- Бизнес-логика в разметке/шаблонах.
- Оставленный отладочный вывод (`console.log`).
- Прямая правка сгенерированных/вендорных примитивов.
- Default export, `any` (если запрещены конвенцией проекта).

## Лимиты размера
- Компонент >150 строк → декомпозиция. Сервис/модуль >100 строк → разбить по ответственности.

## Состояния (UI)
Каждый экран/виджет данных покрывает: `loading`, `error` (+ retry), `empty`, `success`.
Не оставлять «вечный спиннер» и необработанную ошибку.

## Тесты (если есть в проекте)
- Покрывать контракты слоя данных и нетривиальную логику; не тестировать тривиальные геттеры.
