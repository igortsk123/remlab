# Execution Playbook — правила движения по подпланам

> Для исполнителя (Opus 4.8) и владельца. Мастер-план — `plans/MASTER-cost-first.md` (прежний конвейер — архив);
> техядро — MASTER-roadmap (архив). Базовый workflow — `.claude/rules/agent-workflow.md`
> (этот playbook его конкретизирует, не заменяет).

## Старт любой сессии
1. Прочитать `CLAUDE.md` → `.memory_bank/INDEX.md` → `project-state.md` → свой подплан
   (`plans/sub-*.md`) целиком, включая «Прочитай сначала».
2. Не сканировать репозиторий сплошняком — идти по путям из подплана.
3. Если реальность расходится с подпланом (файла нет, код изменился) — НЕ импровизировать
   молча: см. «Отклонения» ниже.

## Железные правила (нарушение = стоп)
- **Код только после «деплой»** от владельца по показанному плану/подплану.
- **VPN-нода `remnanode` на хосте — не трогать никогда.** Правки сервера — только с бэкапом
  и путём отката (`deployment.md`).
- **Секреты только в `.env`** на сервере — не в git, не в памяти, не в логах.
- Файлы вне scope подплана не менять. Нужен чужой компонент — импортировать, не копировать.
- Промпт/модель/шаг пайплайна изменил → bump версии в реестре + трейсинг не отстаёт
  (`.claude/rules/pipeline-tracing.md`).
- Расход на AI: лимиты и дневной $-kill-switch (механика Э0 → М-безопасность) не отключать «чтобы потестить».
- Юридическую логику (ПДн/лицензии) не реализовывать — ставить TODO.

## Чекпоинты владельца «⏸ ПОКАЗАТЬ ВЛАДЕЛЬЦУ»
В подпланах помечены места, где НЕЛЬЗЯ ехать дальше без визуальной проверки владельцем:
скрины экранов, overlay «план поверх фото», разборы `/trace <N>`, таблицы ошибок замера.
Формат показа: 1) что сделано (2–3 фразы простым языком), 2) картинки/ссылки,
3) конкретный вопрос («принимаем? правим X?»). Владелец не кодер — без жаргона.
Особо строгие чекпоинты — в `sub-ml-sizes` (размеры плавают, каждое улучшение сверяется
глазами и рулеткой до перехода дальше).

## Гейты
Каждый подплан завершается машинно-проверяемыми гейтами: `pnpm typecheck && pnpm lint &&
pnpm test && pnpm build` (+ e2e где указано) зелёные + специфичные проверки подплана
(curl, тест-сценарии). Красный гейт = чинить, не «отложим».

## Отклонения от плана (норма, не исключение)
1. Зафиксировать: что ожидалось / что на самом деле / что предлагаю.
2. Мелкое (та же цель, другой путь) → сделать, записать в «Лог выполнения» подплана.
3. Существенное (меняется scope/архитектура/цена решения) → стоп, показать владельцу
   варианты, после решения → `docs/DECISIONS.md` (ADR) + правка подплана.
4. Гипотеза из спеки опровергнута практикой → это ценный результат: в `docs/DECISIONS.md`,
   не замалчивать.

## Выкатка (стандарт, постоянное разрешение владельца 2026-07-02/05)
Завершённое изменение всегда доводится до прода: зелёные проверки → commit → merge в `main`
(ff) → push (CI-гейт) → `./deploy.sh` (бэкап БД + smoke + авто-откат). Прод не оставлять
позади `main`. Предупреждать заранее только о необратимом (миграции с потерей данных).

## Конец сессии / завершение подплана
- Статусы: подплан `in_progress` → `completed` ТОЛЬКО после `/memory-check` + audit чисто
  (гейт из `agent-workflow.md`); частично — `partial`.
- Обновить статус этапа в `plans/MASTER-cost-first.md`.
- Завершённый подплан → `completed_plans/`. `/clear` перед следующим подпланом.

## Эскалация к владельцу (немедленно)
Деньги (тарифы/цены/скидки), удаление данных, миграции БД с риском, всё юридическое,
любые правки сервера вне playbook-путей, расход > дневного лимита, выбор между
архитектурными альтернативами с разной ценой.
