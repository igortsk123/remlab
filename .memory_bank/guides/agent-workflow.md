# Workflow агента — полная версия

Тонкая always-on версия — `.claude/rules/agent-workflow.md`. Здесь — обоснования и детали.

## Почему план перед кодом
Агент с большим контекстом склонен «сразу чинить». Это даёт правки мимо намерения, скрытый
scope-creep и потерю обратимости. План-файл фиксирует намерение, делает scope явным и даёт точку
согласования до необратимых действий. **План обязан быть файлом** — иначе на этапе «деплой»
нечего читать (контекст мог быть очищен).

## Этап 1 — задача → план
1. Прочитать `CLAUDE.md` + `INDEX.md`.
2. По decision tree — нужные Tier 1 сводки; при нехватке — Tier 2.
3. Найти существующие решения/утилиты для переиспользования (не плодить новое).
4. Заполнить `plans/_template.md`: цель, scope in/out, файлы, задачи, критерии приёмки.
5. Сохранить `plans/<slug>.md` (status `draft`). Показать план. Ждать «деплой».

## Этап 2 — «деплой» → выполнение
1. Прочитать план, статус → `in_progress`.
2. Перечислить файлы к изменению. Не трогать остальные.
3. Выполнить минимально достаточно. Прогнать lint/build/тесты, починить.
4. Обновить память — ОБЯЗАТЕЛЬНЫЙ шаг, не «если менялась архитектура» (`memory-discipline.md`):
   затронутые `core/*` (+`updated:`), новая область → `core/<домен>.md`, решения → `decisions.md`,
   смена этапа → `project-state.md` (снимок переписать, хронологию — в `changelog/project-history.md`).
   Затем `/memory-check`; добиться «audit чисто».
5. Полностью → `completed` + дата + summary → перенести в `completed_plans/` (реестры пересоберёт
   аудит). Частично → `partial` (остаётся). Отменено → `cancelled` (остаётся).

### Почему гейт памяти жёсткий
Опыт sup2: шаг «обнови память, если менялась архитектура» в мягкой формулировке не выполнялся
десятки сессий подряд — знание уходило в чат и авто-память харнесса, core/ замёрз на недели.
Условное правило агент трактует как необязательное. Поэтому завершение плана заблокировано,
пока `/memory-check` не выполнен и audit не «чисто» — это единственная точка, где «поработал →
не записал» ловится механически (PLAN-MISPLACED/PLAN-STUCK в audit подстраховывают).

## Управление контекстом
- `/clear` перед новой задачей — чистый контекст точнее следует памяти.
- После ~3 PR подряд предложить `/clear` (накопленный контекст начинает мешать).
- Если контекст перегружен ещё до плана — попросить `/clear`, затем планировать.

## Изоляция фич
Ветки `feature/<area>`; не редактировать файлы чужой фичи; нужный чужой компонент — импортировать.
Это снижает конфликты и держит PR обозримыми.
