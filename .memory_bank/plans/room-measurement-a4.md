## Цель
Замер комнаты по **одному фото** с листом A4 на полу. Пользователь текстом даёт **рост (обязательно)** и
**модель телефона (желательно)**. На выходе — **одно «замерщицкое» фото**: все основные размеры (комната Д×Ш,
высота потолка, окна В×Ш, эркер Ш×Г×В, двери) нанесены аккуратно, как от живого замерщика, чтобы по ним
подбирать мебель/материалы. Точность: пол/габариты 5-7%, вертикаль на калиброванной стене 7-12%, сложные
элементы (эркер) ~10% при видимых углах; перекрытое — честный флаг «≈, уточнить», не выдумываем.

## Статус эксперимента (что уже ПОДТВЕРЖДЕНО на реальном фото)
- OpenCV сам детектит A4 (Otsu+contours, reproj err 0.76 px), `solvePnP` восстанавливает позу камеры.
- Замер по полу = **199.5 см против рулетки 200 → ошибка 0.3%**.
- FOV подтверждён с 3 сторон: A4-геометрия, рост человека (высота камеры 153.5 см), **паспорт телефона**
  (iPhone SE 3 = 28 мм → портретный верт. FOV 63.4°; наш калибр 62° — совпадение в пределах ~1.4°).
- Инфра: раздача `remont-lab.online/lab/*` → `/opt/remlab/temp` уже поднята (см. `deployment.md`).

## Ключевые решения (этой сессии, 2026-07-05)
1. **Входы от пользователя текстом:** рост — обязателен; модель телефона — желательна (точнее FOV);
   опц. подсказка потолка. Без сложного UI — простые текстовые поля.
2. **Источник FOV (порядок):** (a) **таблица «модель телефона → FOV»** — лучший, точный, простой;
   (b) EXIF оригинала; (c) калибровка по росту (дожать); ~~(d) обобщённая таблица «типичный телефон» — выкинуть~~.
3. **Меряет всегда OpenCV** (детерминизм), **что/где мерить — Vision-LLM**, рисует — рендер-шаг, LLM
   причёсывает подписи и НИКОГДА не пересчитывает числа (принцип Cost Engine).
6. **Двухступенчатая разметка (2026-07-05, важно):** Gemini = СЕМАНТИЧЕСКИЙ ДЕТЕКТОР ОРИЕНТИРОВ — отдаёт
   ТОЛЬКО грубые seed-точки (где A4/стены/окна/линия пола). **OpenCV сам УТОЧНЯЕТ** контуры локально в ROI
   вокруг seed'ов: cornerSubPix (углы A4), Canny+HoughLinesP (линии пол↔стена, стена↔потолок, вертикали окон),
   fit границ пола/потолка. Gemini НЕ финальный измеритель и НЕ источник точных пикселей — только «где искать».
   Это снимает вариативность Vision (seed гуляет ±20px → уточнение притягивает к реальному ребру).
   Урок из run-2: Gemini дал «контур пола» на высоте подоконника → мусор; фикс = (а) промпт: floor_wall_edge
   строго у плинтуса; (б) OpenCV-уточнение линии в ROU, а не доверие пикселям Gemini.
4. **Каждый этап пишет артефакт в `/opt/remlab/temp` → ссылка `remont-lab.online/lab/run-<id>/...`**,
   чтобы на тесте видеть все промежуточные результаты (текст+фото), не только финал.
5. **Фото — оригиналом** (не через мессенджер): иначе режется EXIF и кадрируется, FOV плывёт. Предупреждать.

## ВНЕШНЕЕ ИССЛЕДОВАНИЕ (GPT deep research, 2026-07-06) — выводы для внедрения
Отчёт REMLAB прогнан через GPT deep research (ждём ещё Claude). Совпал с нашим на ~90%. Ключевое:
- **ДВА ПРОДУКТА:** free = 1 фото (видимое + честный confidence), paid = 3–5 фото/видео (multi-view
  VGGT/MASt3R/DUSt3R + A4-якорь; GPU-воркер). Не выдавать «Д×Ш×В комнаты» как измеренное по 1 фото.
- **Instance-сегментация вместо semantic** (YOLO-World/Grounding DINO → **SAM 2** / Grounded-SAM 2). SegFormer
  ADE20K — только stuff (пол/стена/потолок/окно). Решает потерю правого стула/стола. ← фикс №1.
- **Confidence-таксономия:** measured / estimated / not_visible / need_extra_photo — на каждый размер.
- **Окна:** wall-plane + opening polygon + frame/mullion, ФИЛЬТР ЖАЛЮЗИ (периодические тонкие горизонтали),
  проекция на стену НЕ на стекло (прозрачность ломает depth — подтверждено литературой). Створки — только при
  высокой уверенности, иначе «остекление целиком».
- **Depth-ансамбль:** Depth Anything V2 metric + **Metric3D v2** (+ surface normals); расхождение = uncertainty.
  Горизонталь по полу — приоритет ray-plane (solvePnP), не raw depth. Вертикаль — wall-plane. Depth для объектов —
  фильтр фона/кластеризация, не единственный измеритель.
- **Room-layout сети (RoomFormer/RoomNet/NonCuboidRoom) — НЕ источник чисел**, только гипотеза скрытой линии
  пол↔стена, проверяемая метрическим слоем. RoomFormer — для 3D-сканов, не для 1 RGB.
- ⭐ **DATASET HARNESS (30–50 комнат: фото+A4+рулетка+потолок+3–5 контрольных) — приоритет №1.** Иначе
  переобучаемся под один room1.jpg. Наш акцент: поставить ВПЕРЁД.
- Мой пушбэк: **CPU-латентность free-тира** (SAM2+GroundingDINO+2 depth на CPU может не влезть в UX ~1.5мин)
  — замерить, возможно лёгкие модели/async/GPU-воркер уже во free.
- Референс рынка: серьёзный замер (RoomPlan/magicplan/Canvas) = LiDAR+AR, не 1 фото. Наше УТП — дешёвый
  предварительный обмер по обычному фото + «доснимите 3–5 кадров» для точного.
- Приоритет внедрения: 1) dataset harness, 2) instance-seg, 3) confidence, 4) window-модуль, 5) depth-ансамбль,

## РАБОТА С ПЕРЕКРЫТИЕМ — мебель ВСЕГДА закрывает низ (owner, 2026-07-05, КРИТИЧНО)
Реальные фото всегда с мебелью → низ стен/пола перекрыт на 50%+. «Чистого кадра» НЕ будет. Наивная «верхняя
граница маски пола» проваливается (мерит по низу мебели). Принцип: **МЕРИТЬ СВЕРХУ, где не перекрыто.**
Мебель стоит на ПОЛУ, не на ПОТОЛКЕ → потолок и верх стен видны почти всегда.

1. **Габариты комнаты — из ПОТОЛКА, не из пола.** Линия стена↔потолок (не перекрыта) → луч камеры ∩ плоскость
   `Z=потолок` (потолок — вход пользователя) → углы комнаты в XY. Стены вертикальны → footprint пола = тот же.
   Перекрытие пола больше НЕ мешает. Это главный разворот.
2. **Вертикальные углы стен** видны выше мебели → экстраполировать вниз до пола (даже если угол у пола закрыт).
3. **Room-layout модель** (ST-RoomNet / RoomFormer / non-cuboid) — предсказывает линию пол↔стена СКВОЗЬ мебель
   (amodal). Заменяет наивную границу маски пола. Опция для точной формы.
4. **Окна/двери** обычно выше мебели → маски (SegFormer) + плоскость стены, калибр по потолку.
5. **Мебель (footprint)** — маски SAM + ВИДИМЫЕ точки контакта ножек с полом, НЕ bbox (bbox геометрически врёт).
6. **A4** — абсолютный масштаб (мелкий, на полу, обычно виден). Потолок(вход) — вертикальная линейка.
7. **Фьюжн + confidence:** сверять floor-оценку и ceiling-оценку; конфликт → флаг, брать надёжную.

Итог: источник габаритов комнаты — ПОТОЛОК/верх стен + известная высота потолка + A4-масштаб. Низ (пол/мебель)
— только там, где реально видно. Наивный floor-mask-boundary (run-5) — DEPRECATED для формы комнаты.

**ПОДТВЕРЖДЕНО НА room1 (metric-depth, 2026-07-05):** Depth Anything V2 (metric indoor) + якорь масштаба по A4 →
потолок вышел **264 см (эталон 263)** сквозь заставленную комнату, пол на Z≈0, габариты по стенам+потолку
**343×286 см** (наивный пол-метод давал искалеченные 253×241). Точечные floor-дистанции у depth шумнее (±15%,
A4→стол 229 vs 200) → solvePnP+A4 точнее. **РЕШЕНИЕ бесплатного тира = ГИБРИД:** масштаб+пол = solvePnP+A4,
высоты+стены+скрытое = metric-depth. Всё локально CPU $0. Модель: `depth-anything/Depth-Anything-V2-Metric-Indoor-Small-hf`

## КАНОНИЧНАЯ АРХИТЕКТУРА (owner, 2026-07-05) — 4 роли, НЕ смешивать
Финальный замер — это НЕ задача Gemini и НЕ совсем задача OpenCV, а прикладная геометрия + бизнес-логика.

1. **Gemini #1 — размечает сцену (семантика).** «Это лист / окно / правая стена / стул / эркер». Грубые
   seed-точки в пикселях. НЕ меряет, точность до пикселя не нужна.
2. **OpenCV — уточняет пиксельную геометрию.** «Вот точные углы / линия / контур / homography». В ROI вокруг
   seed'ов: cornerSubPix/Canny/Hough; сам детектит A4; solvePnP → поза; лучи∩плоскости. НЕ понимает интерьер,
   НЕ принимает смысловых решений. Отдаёт сырую геометрию (точные пиксели + 3D).
3. **Python-модуль — «мозг расчёта» (решения + размеры + бизнес-логика).** Именно он решает: «эта линия —
   задняя стена», «ширина проёма ≈220 см», «глубина эркера ≈55 см», «высота потолка 250–270», «понизить
   confidence», «результат конфликтует с потолком 2.5 м». Детерминированный, как Cost Engine. ЗДЕСЬ рождаются
   числа. OpenCV даёт инструменты — модуль принимает решения.
4. **Gemini #2 — интерпретирует и красиво объясняет.** Готовые числа → подписи/сводка/риски для человека.
   НИКОГДА не пересчитывает (use_only_module_numbers).

Поток: фото → **Gemini#1 (разметка)** → **OpenCV (уточнение+геометрия)** → **Python-модуль (расчёт/решения)**
→ рендер фото + **Gemini#2 (объяснение)**. (В нашем стеке «GPT Vision» из обсуждения = Gemini.)

## Пайплайн — этапы (кто что делает, что на выходе, какая ссылка)

**Stage 0 — Приём и параметры съёмки**
- Вход: фото + рост + модель телефона (+ опц. потолок). Прочитать EXIF (модель/фокусное), если есть.
- Разрешить FOV по порядку решения №2. Зафиксировать источник и уверенность.
- Артефакт: `00_input.json` (входы, выбранный FOV + источник), копия фото. → `/lab/run-<id>/00_input.json`

**Stage 1 — A4 + поза камеры (OpenCV)**
- Автодетект A4, уточнение углов (Canny/contours/approxPolyDP, ratio≈1.414).
- Камера-матрица из FOV → `solvePnP` (IPPE) → высота, наклон.
- Self-check: reprojection error; если высота далеко от прайора роста → пересчитать FOV (калибровка).
- Артефакт: `01_camera.json` (поза, FOV, reproj, confidence) + `01_a4_detected.jpg`. → ссылки.

**Stage 2 — Поиск объектов (Vision-LLM: OpenAI/Gemini)**
- Vision находит что мерить: стены, окна, двери, **грани эркера**, линию «стена-потолок», footprint пола.
  Возвращает черновые пиксельные углы + `plane_id` + `touches_floor`/`same_plane_as` + confidence.
- Артефакт: `02_objects.json` + `02_objects_overlay.jpg` (черновые рамки). → ссылки.
- До интеграции Vision — углы задаём вручную/сидом (как в спайке), формат тот же.

**Stage 3 — Геометрия и замер (OpenCV)**
- Плоскость пола из позы. Для каждого объекта:
  - расстояния/footprint по полу → пересечение лучей с полом;
  - **вертикаль на стене** → «линейка» по известному потолку (2.63 у теста) + линия «стена-пол»;
  - **эркер** → углы «стена эркера ∩ пол» дают ширину/глубину/углы граней (пол эркера = та же плоскость).
- На каждый размер: значение см, error %, confidence, `needs_user_confirmation` (перекрытие/низкая увер.).
- Артефакт: `03_measurements.json`. → ссылка.

**Stage 4 — Отрисовка «как замерщик» (рендер)**
- Нанести все размеры: выносные линии, стрелки, подписи со смещением, без наложений; стиль по уверенности
  (сплошная — точно, пунктир + «≈» — оценка/перекрыто). Это ФИНАЛЬНЫЙ артефакт.
- Артефакт: `04_final_surveyor.jpg`. → ссылка (главный результат).

**Stage 5 — Интерпретация и сводка (LLM, опц.)**
- Краткое резюме, таблица размеров, предупреждения, заметки под подбор мебели. `use_only_opencv_numbers`.
- Артефакт: `05_summary.json` + `index.html` (фото+таблицы+ссылки на все этапы). → `/lab/run-<id>/`.

**Тест-обёртка (сейчас, для эксперимента)**
- Один прогон = папка `/opt/remlab/temp/run-<id>/` со всеми артефактами + `index.html`, который линкует

## Продуктизация (Фаза E) — архитектура
- **Роли (контракт):** LLM ищет/объясняет, OpenCV считает; числа не из LLM.
