## Цель
Сделать так, чтобы **выбор пользователя реально управлял генерацией**. Двухэтапный флоу: сначала
разбираем фото на реальные объекты и предлагаем действия, пользователь правит (поменять/оставить/убрать),
и только потом генерируем картинку по ЕГО решениям + стилю.

## Источник задачи
Разбор владельца по генерациям №1/№3: сейчас пользователь жмёт заранее заданные кнопки, но
(а) по объектам решает модель, а не пользователь, (б) промпт перерисовки игнорирует и объекты из analyze,
и кнопки «что оставить», и ограничения — картинка это общий рестайл по стилю. Нужно, чтобы решения
пользователя доходили до генератора.

## Как сейчас (факт, из кода) — что меняем
- `modules/generation-job` (`runPreview`) делает analyze → restyle → ideas за один заход, без остановки.
- `lib/prompts/registry.ts` `restylePrompt.build({style, brief})` — использует ТОЛЬКО стиль + interventionLevel.
  Игнорирует `analysis.objects`, `brief.keep`, `brief.constraints`.
- `analysis.objects` (keep/suggest_change) решает модель; влияет только на текст ideas (и то через summary).
- Экран `app/p/[id]/preview` показывает analysis-чипсы и ideas ПОСТФАКТУМ.

## Целевой флоу (двухэтапный)
**Этап A — «Разбор» (analyze, интерактивный):**
1. Пользователь загрузил фото + базовый бриф (стиль, бюджет, ограничения — как сейчас).
2. Vision-модель по ЭТОМУ фото возвращает список **реальных объектов** + предложенное действие на каждый:
   `change` (поменять) / `keep` (оставить) / `remove` (убрать). (Сейчас только keep/suggest_change → добавить remove.)
3. Показываем пользователю экран выбора: объекты с переключателями действий, редактируемо
   (можно переопределить предложение модели), + подтверждение стиля.

**Этап B — «Генерация» (restyle по выбору пользователя):**
4. Пользователь подтвердил. Собираем промпт из: фото (референс) + **явные пообъектные инструкции**
   (что поменять / оставить нетронутым / убрать) + выбранный стиль/палитра/уровень.
5. Генерируем картинку. Показываем результат (+ ideas/каталог/бюджет как сейчас).

## Скоуп — что входит
- Разбить `runPreview` на два шага с точкой ожидания выбора пользователя между analyze и restyle
  (сейчас всё в одном вызове). Вероятно: `runAnalyze(projectId)` и `runGenerate(projectId)`.
- Расширить `analysis.objects.action`: добавить `remove` (contracts/project + room-analysis + промпт + UI).
- Хранить выбор пользователя по объектам в проекте (новое поле, напр. `objectChoices`/расширить analysis).
- Переписать `restylePrompt.build` — принимать пообъектные инструкции пользователя и вшивать их явным
  списком «поменять: …; оставить без изменений: …; убрать: …» + стиль. Бампнуть `version`.
- Новый/расширенный экран выбора объектов (между брифом и превью) + серверный экшен сохранения выбора.
- Обновить `app/p/[id]/preview` и прогресс-степпер под новый шаг.
- Трейсинг: новый состав шагов → бампнуть версию пайплайна (`lib/pipelines/registry.ts`), промпты — version.

## Скоуп — что НЕ входит (пока)
- Реальный подбор товаров из фидов (отдельный воркстрим v0.3, ADR-0014).
- Инпейнтинг точечных объектов (замена конкретного предмета «на месте») — сложнее; пока перерисовка всей
  комнаты, но управляемая пообъектными инструкциями.
- Изменение бизнес-модели/paywall.

## Решения человека (нужны от владельца — не Claude)
- [ ] Три действия на объект достаточно (**поменять / оставить / убрать**) или нужно иначе?
- [ ] Что делает «поменять» на объекте, если стиль общий: менять в рамках выбранного стиля? Дать ли выбор
      «на что похоже» (варианты)? (Влияет на промпт и экран.)
- [ ] «Готовые варианты изменения» в том же экране (как вы предложили) — сколько и какие пресеты?
- [ ] Оставляем ли кнопки брифа «что оставить» (дублируют пообъектный выбор) или убираем ради одного места правды?
- [ ] Порядок экранов и тексты (дизайн — за владельцем).
- [ ] Приемлемо ли, что это 2 обращения к модели с паузой (чуть дольше, чуть дороже на разбор).
