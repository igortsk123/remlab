# Stage 1 — Master Roadmap (верхнеуровневый план)

> Верхнеуровневый план всего Stage 1. Каждый блок ниже (M0…M8) исполняется как **отдельный
> под-план** (`plans/<slug>.md`, статус draft → «деплой» → completed), строго по порядку.
> Цель этапа: **работающий сквозной срез продукта с настоящим ИИ** — фото → стиль → реальное
> AI-превью → ограниченный бесплатный результат → paywall(заглушка оплаты) → «Мои комнаты».

## Что изменилось vs первичный черновик
Появился доступ к ИИ, поэтому НЕ делаем всё на заглушках — настоящий ИИ подключаем внутри Stage 1:
- **Текст/зрение (анализ комнаты, идеи, подбор):** OpenAI/ChatGPT — ключ берём из соседнего
  проекта `../v0-health-card/.env.local` → копируем в `remlab/.env` (в git НЕ коммитим).
- **Генерация картинок:** Google Gemini **`gemini-3.1-flash-image`** (Nano Banana 2), endpoint
  `https://generativelanguage.googleapis.com/v1beta`, заголовок `X-goog-api-key`. Ключ → в `.env`.
  Проверено 2026-07-01: ключ рабочий, модель в списке доступных. Значение ключа — только в `.env`.
- Доступные image-модели с ключом: gemini-3.1-flash-image (осн.), gemini-3-pro-image (тяжелее),
  imagen-4.0-* (запас). Финальный выбор модели фиксируем ADR после спайка (M0).

## Порядок под-планов (идём подряд)

### M0 — Провайдеры и ключи (фундамент ИИ)
- `lib/providers/`: клиент Gemini-image + клиент OpenAI-text/vision за общим интерфейсом
  (`Result<T,E>`, ретраи, таймауты). Провайдер прячется за интерфейсом модуля.
- Ключи в `.env` (локально + сервер), `.gitignore` проверить; запись «где ключ» в
  `core/access-and-integrations.md` (значения НЕ писать).
- Спайк: 1 реальная генерация картинки + 1 реальный vision-вызов (скрипт/тест).
- ADR: выбранные модели и почему.

### M1 — Данные (схема БД + контракты)
- Drizzle-схема ядра (users, properties, rooms, room_projects, result_versions,
  style_profiles, uploaded_images, detected_objects, generation_jobs, generated_images,
  product/material previews) + миграции (обратимые).
- Zod-контракты в `/contracts` (анти-дрейф, `z.infer`). Anonymous session (cookie → users).

### M2 — Каркас flow (7 экранов «проходятся насквозь»)
- `app/(flow)`: landing → комната/цель/уровень → фото/бриф → style cards →
  AI-preview → paywall → workspace. Нейтральный дизайн, состояние в БД.
  AI-блоки пока подключены как заглушки — заменим в M3/M4.

### M3 — Реальный анализ комнаты (OpenAI vision)
- По фото: распознать объекты (диван/окно/пол/шкаф…), понять комнату; из style cards собрать
  `style_profile`. Экран «AI понял комнату» показывает реальные данные.

### M4 — Реальное визуальное превью (Gemini Nano Banana)
- Фото комнаты + style_profile → сгенерированная картинка «как станет».
  Долгий вызов НЕ в HTTP-роуте: job-строка + фон (Inngest или простой polling статуса).
  Честное предупреждение «это концепт, не рабочий проект».

### M5 — Бесплатный результат: идеи + превью товаров и материалов
- 3–5 идей (LLM). Ограниченный preview: 2–3 **товара** (мебель/декор/свет/текстиль) И
  2–3 **материала** (краска/обои/покрытие) — «показано 3 из 18». Каталог на этом шаге — seed/заглушка
  (реальные источники — отдельный поздний план, решение владельца по источникам).

### M6 — Paywall + разблокировка room pack
- Экран оплаты со структурой «что внутри полного пакета». Оплата пока **заглушка**
  (кнопка «оплатить» → разблокировка), реальная YooKassa — поздний план.

### M7 — Workspace «Мои комнаты»
- Объект → комнаты → версии; сохранение результата, создание новой комнаты/версии.

### M8 — Закалка (observability + тесты + деплой)
- Sentry + PostHog + события flow. Playwright: happy-path + ≥1 путь ошибки (плохое фото).
- CI: шаг `db:migrate`. Деплой на exit-fi (бэкап+rollback, VPN не трогаем).
