# Коммерческий мастер-план — «план из планов» для конвейера «Деньги»

> Спина коммерциализации. Каждый этап при подходе исполняется по своему подплану
> (`plans/sub-*.md`, реестр ниже). Исполнитель — Opus 4.8 под сопровождением владельца;
> правила движения — `guides/execution-playbook.md`. Техническое ядро measured-fit живёт в
> [[MASTER-roadmap]] (Ф0–Ф8) — этот план его НЕ дублирует: Э5 здесь = Ф8 там.
> Бизнес-истина — `docs/master-brief-v0.3.md` (ADR-0014).

## Зачем (итог аудита 2026-07-11)
Прод живой, ops зрелый (CI/deploy/трейсинг), но выручечных механизмов в коде ноль: каталог —
seed 8 позиций, оплата — заглушка 490 ₽, фидов/реф-ссылок/платежей нет; ядро «влезет/не влезет»
вне git и не интегрировано; спрос деньгами не проверен (Stage 0 брифа не начат).
Оценки: идея 8/10, текущее состояние 6/10. Полный разбор — сессия 2026-07-11.

## Принцип
Два конвейера параллельно. **Деньги:** валидация руками → фиды → платежи → рост.
**Ядро:** стабилизация замера → интеграция fit ([[MASTER-roadmap]]).
Деньги не ждут ядра; ядро подключается как усилитель конверсии по замеренному спросу.
Каждый этап = timebox + числовой gate + kill-критерий. Код пишется только под то,
что руками подтвердило спрос.

## Сквозной kill-критерий проекта (из брифа)
CAC покупателя > 2× дохода после двух итераций каналов и оффера (конец Э4) →
остановка/пивот, а не «ещё одна фича». Решение — владельца.

## Первые 2 недели — три вещи
1. Гдеслон + измерение денег (аккаунт, ручные реф-ссылки с subid, воронка до «клика по товару»).
2. Стоп-кран: mltest в git, дыра владения, лимиты + $-kill-switch.
3. Один прайс (fake-door 1490/9900) и первые 10–20 тыс.₽ трафика.

## Что СОЗНАТЕЛЬНО не строим до валидации
Cost Engine/чертежи/PDF (продаётся как concierge в 9900); подписки (только разовые платежи);
auth сверх magic-link; i18n/UK; видео-тир; апгрейды замера сверх минимума; инжест всего фида
(только целевые категории); B2B-тулинг; eval сверх минимального контура.
