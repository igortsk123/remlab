# Цели: замер + меблировка/замена по ОДНОМУ фото (fit-aware)

> Сформулировано владельцем 2026-07-06. Это цели/направление, не план исполнения (план — `plans/room-measurement-a4.md`).
> Tier 1 сводка: `core/goals.md`.

## Продуктовая суть — два флоу
- **Одно фото (СЕЙЧАС):** меблировка / замена / подбор в ВИДИМОЙ части комнаты. Пользователь снял половину комнаты, хочет что-то поменять/обставить.
- **10-сек видео (ПОЗЖЕ):** ремонт / полный план комнаты (multi-view, сквозь мебель, зона за камерой).

Флоу одного фото: загрузил фото → детектим объекты → «что хочешь заменить/изменить?» → выбирает (напр. «диван»).

## ГЛАВНЫЙ КОЗЫРЬ — measured fit (отличие от First Chair и др.)
Конкуренты (First Chair и пр.) **вставляют товар ВСЛЕПУЮ**: в область непонятного размера, диван может быть вдвое больше зоны — пользователь не знает, влезет ли, пока не поедет мерить.

**Мы:** знаем И размер зоны/комнаты, И размер товара → можем сказать **«влезет / не влезет»** и вставить в **ПРАВИЛЬНОМ масштабе** + посоветовать «убери стол — тогда встанет».
- Ценность: **застройщики** (продажа планировок квартир) + **дизайнеры** → шире рынок. Это оправдывает борьбу за точность ±10%.
- Владелец: ±10% — приемлемо («диван 1м, система скажет влезет в 1.10 — прощу»). НЕ приемлемо: зона 1м, а система пихает диван 1.5м (это и делает First Chair).

## Целевой конечный результат (что продукт должен ГОВОРИТЬ)
- «Диван 1.5 м не влезет в зону 1.2 м — велик на ~30 см. Предлагаю: (а) сюда — диван до 1.2 м; (б) убрать стол+стулья → тогда 1.5 м встанет вот здесь.»
- + визуализация в правильном масштабе («вот так впишется» / «торчит»).

## Три механики (уточнено 2026-07-06)
- **A — заменить ОДИН объект + fit-check:** detect → маска → вырезать (LaMa) → сверить размер товара с
  **footprint зоны ПОД объектом** → вставить в масштабе (SD-inpaint) → **вердикт влезает/велик на N см**.
  Мерим: footprint убранного объекта (+ соседний свободный пол если новый больше).
- **C — ОСВЕЖИТЬ всю мебель (сильный affiliate-кейс):** спросить «это living room? что примерить из каталога?»
  → убрать всю/выбранную мебель (маски+инпейнт → пустая комната) → **замерить ВСЮ комнату** (площадь пола,
  стены, высота, Д×Ш) → вписать новый НАБОР из каталога (диван+стол+стулья) с ±10% → полировка (fal) → показать
  с размерами + вердикт «влезает / впритык, проверь». Мерим: **вся комната**, НЕ свободная площадь.
- **Видео-тир — ремонт/полный план:** multi-view сквозь мебель, вся геометрия квартиры.

Важно: свободная площадь (зелёный пол) — только для режима A (куда встанет новый предмет). Для режима C нужна
**площадь/габариты всей комнаты**. За убранной мебелью пол ДОСТРАИВАЕТСЯ инпейнтом (угадан, не истинный замер);
полная геометрия комнаты — видео-тир.

## Что нужно для «влезет / не влезет»
1. **План пола:** стены + прямоугольники «занято» (мебель) + **свободные зоны** (считаем из замера: свободный пол ~6.1 м² на room1, футпринты объектов).
2. **Размеры зоны:** footprint убранного объекта (для замены на месте) + свободный пол (для новой расстановки).
3. **Каталог:** размеры товаров Ш×Г×В (данные готовим).
4. **Движок размещения:** прямоугольник товара vs свободная зона + зазоры на проход → да/нет + позиция; «что убрать, чтобы влезло» = пересчёт зоны после удаления.
5. **Визуализация в масштабе** (fal): товар нужного пиксельного размера в кадр.

## Технический стек (сошлись по ходу экспериментов)
- **Масштаб/поза:** лист A4 + OpenCV `solvePnP` (сверено рулеткой: A4→стол 199.5 vs 200 = 0.3%). FOV из модели телефона + авто-калибровка по росту.
- **Instance-детекция мебели:** torchvision **Mask R-CNN** (лицензия BSD, CPU ~2.4с) — оба стула/стол/матрас РАЗДЕЛЬНО (SegFormer сливал). НЕ Ultralytics YOLO (AGPL).
- **Глубина:** Depth Anything V2 metric + якорь A4 (потолок 261 vs 263 = 0.8%). Апгрейд: Metric3D / Depth Large через fal.
- **Stuff (пол/стена/потолок/окно):** SegFormer ADE20K (маски + свободный пол).
- **Уточнение масок:** SAM 2 через fal (точные границы → точный footprint/зона).
- **Инпейнт «вырезать»:** **fal.ai LaMa** — работает, чисто (ковёр восстанавливается, остальное пиксель-в-пиксель).
- **Вставка/полировка:** fal.ai SD-inpaint / Flux по маске (замок на остальное — не перегенерим всё).
- **Оплата инфры:** fal.ai pay-per-request (UK-карта владельца, $0 в простое, вкл/выкл не нужен). Масштаб — RunPod Serverless.

## Честные ограничения (принять и заложить в UX)
- **Одно фото = только ВИДИМАЯ зона.** «Влезет тут» — да; «перенеси на дальнюю стену / полный план» — нужен **видео-тир** (перед за камерой не виден).
- **Точность ±10%** на хороших осях (пол/зона/вертикаль). Пограничные случаи метить «впритык, проверьте».
- Глубина слабее по горизонтали (±14%) и сквозь стекло → пол считаем solvePnP, вертикаль — depth, окна — по стене/раме не по стеклу.
- **Перекрытие:** «убрать стол, чтобы влез диван» — моделируем (удалили → пересчитали свободную зону).
- Числа рисует наш детерминированный код; image-gen НЕ рисует цифры (исказит).

## САМО-ПРОВЕРЯЮЩИЙ движок (owner 2026-07-06) — работать на ЛЮБОМ фото, не подгонять под одно
Принцип: НЕ хардкодить под конкретное фото. Тестируем на разных → находим ОБЩИЙ баг → чиним алгоритмически
(пример: дверь на угловой стене занижалась вдвое → метод «4 угла + основание на полу + вертикальная плоскость
стены» → высота 192 vs эталон 196 = 2%; теперь работает для любого углового объекта).
4 механизма движка:
1. **Многоалгоритмовый замер** каждой величины (solvePnP-геометрия / depth / прайор) → берём согласованный.
2. **Таблица типовых размеров** (плаузибилити-приоры): стул сиденье 42-48 / общая 85-110; стол Ø90-130 h72-78;
   дверь-полотно 60-90×195-205; диван 180-240×85-100×75-90; подоконник 80-95. Как sanity-check + fallback.
3. **Само-проверка (self-check):** кросс-метод (разошлись→флаг); плаузибельность (вне таблицы→перемерить другим
   алго); геометрия (объект не выше потолка; сумма стен=периметр; footprint на полу).
4. **Fallback-цепочка + confidence/provenance** на каждый размер (измерено / кросс-проверено / по-приору / флаг).
Движок сам гоняет проверки до сходимости, выбирает метод под ситуацию. Настраивается ТОЛЬКО на датасете разных
комнат (не на одном фото).

## ЧЕЛОВЕК ПРАВИТ ЦИФРЫ (human-in-the-loop) — снимает нагрузку на точность
Отдаём фото с выделенными объектами + предзаполненные редактируемые поля на каждый: `Ширина: __ Высота: __
Глубина: __` (наши цифры + confidence уже вписаны). Человек корректирует + отмечает что заменить/изменить.
Правки → в fit/placement. Формула: 10% автозамер + правка человека = надёжно (честнее, чем притворяться точным).
UX: клик по объекту на фото → всплывают поля Ш/В/Г/Глубина.

## ИСТОЧНИК словаря детекции + таблицы типовых = ФИДЫ РИТЕЙЛЕРА (owner 2026-07-06, не ручной список)
Ручная таблица типовых размеров (в `mltest/sizes_prior.py`) — временный костыль для проверки логики Ф3.
Продакшен: и **словарь open-vocab детекции** (Grounding DINO), и **таблица типовых размеров** (плаузибилити-приоры
для само-проверки) строятся из **фидов Гдеслон** (реальные Ш×Г×В тысяч товаров по категориям → перцентильные
диапазоны). Один источник закрывает оба. Плюс фикс-стандарты для архитектурных элементов, которых нет в мебельных
фидах (дверь 200×80, потолок 250-270, подоконник 80-90). Преимущества: полнота, реальные цифры (не оценки),
само-обновление, согласованность (сверяем замер с тем же каталогом, из которого подбираем товар).

## ПОКАЗЫВАЕМ ТОЛЬКО КРУПНОЕ (owner 2026-07-06) — мелочь детектим, но не рисуем
Пользователю на фото — только крупные объекты (мебель/приборы/окно/дверь) с размерами. Мелочь на столе/диване
(чашка/бутылка/клавиатура/ноутбук) детектим для полноты, но НЕ рисуем отдельными боксами — сворачиваем в
«+предметы:N» на опоре (иначе фото шумит). OpenAI-именование незнакомого — только для КРУПНЫХ незнакомцев,
маленьким кропом (~256px, доли цента), 0-2 на комнату; полное фото не шлём, текст не называет по пикселям.

## ПРАВИЛО: не исправлять детектор кодом — честный перевод + правка человеком (owner 2026-07-06)
Если английская модель детекции (Grounding DINO) ошибается — НЕ строить сложную логику авто-исправления
(таксономия-форс, vision-переименование кропов). Вместо этого **дословно переводить сырой ярлык DINO через OpenAI**
(«wardrobe door»→«дверь шкафа», «office chair»→«офисный стул»; чистим только мусор токенизации ##/дубли-синонимы).
Ошибки поправит человек в UX-правке. Таксономия остаётся только для ВНУТРЕННЕЙ классификации (kind опора/пол/стена
+ таблица размеров), не для показа. Само-проверка помечает неправдоподобное красным (напр. пуф, помеченный «диваном»).
Причина: точность детекции добирается человеком дёшево; авто-исправление дорого (vision-кропы) и всё равно ошибается
под оклюзией (радиатор за стулом → vision видел стул). Реализация: `mltest/openai_translate.py` (faithful-промпт),
`run_f3.py` (подпись=translate(raw)); `openai_vision_name.py` отключён.

## КАСКАД замера (owner 2026-07-06) — ступенчато, цель «дотянуть достоверность» = Single View Metrology
Owner: относительные измерения по максимуму — надёжное сначала, потом им подтягиваем остальное. Академ.основа
подтверждена (Criminisi «Single View Metrology» 2000): объект известного размера = линейка; распространение 3D от
объектов на полу к остальным. Ступени (в `mltest/run_f3.py`):
1. **Надёжное:** напольные — высота ГЕОМЕТРИЕЙ (основание на полу + луч на верх, БЕЗ depth: `h_geom`) — надёжнее depth,
   даёт правильный ОТНОСИТЕЛЬНЫЙ порядок (стул>радиатор). Общая калибровка пола делает все высоты взаимно согласованными.
2. **Консенсус:** одинаковые классы (стулья, радиаторы) → медиана, выброс подтянуть.
3. **Расхождение способов:** окно/дверь мерим 2 способами (плоскость depth vs фронталь); разошлись >28% → НЕ зелёное,
   «(способы расх.)» жёлт (окна эркера так и помечены — больше не врут зелёным).
4. **Каталог+потолок:** сверка (avg±tol=ok / между=коррекция / вне min-max=ошибка) + верх окна/двери ≤ потолок.
5. **Эталон ⚓:** conf≥0.85 + способы сошлись → пометка (надёжный якорь масштаба).
ЧТО НЕ РАБОТАЕТ (убрано): кросс-сравнение РАЗНЫХ объектов по пиксельному отношению — врёт при неточных рамках
(стол→122, дверь→269). Надёжный относительный сигнал даёт только общая геометрия (`h_geom`), не пиксель-отношения.
Результат: стулья 104-109 (контроль 108) ✓, стул>радиатор ✓; окна эркера жёлтые «проверить» (числа 165/104 всё ещё
завышены — нужна эркер-геометрия угловых стен). Файлы: `run_f3.py`, `object_catalog.py`, `sizes_prior.py`, `taxonomy.py`.

## НАХОДКА: footprint-ГЛУБИНА из 1 фото ненадёжна (2026-07-06) → Ш+позиция надёжно, Г=каталог/видео
Пробовал мерить реальную глубину footprint через SAM2-маску (нижняя кромка/полоса → пол → minAreaRect): из ОДНОГО
фото ненадёжно — зад объекта перекрыт, проекции скачут (стол 45↔126, матрас Г 20↔61 от полосы к полосе). Откатил.
Надёжный сигнал из 1 фото: **ширина + позиция footprint** (проекция низа рамки на пол через A4/solvePnP). **Глубина Г —
оценка из каталога** (для «заменить на такой же» ок). Истинная Г — **видео-тир** (мультивью, зад виден). Это согласуется
с «одно фото = только видимое». Модуль `mltest/run_plan.py` — план пола сверху: комната+пол(м²)+footprint мебели+свободное.
room1 ~249×313 6.2м², room2 ~230×306 5.2м². Фильтры: только на видимом полу (внутри контура), двери/окна исключены.

## Приоритеты внедрения
1. Довести **замер зоны/комнаты** (SAM2 + Depth Large через fal) → точнее план и свободный пол.
2. **Fit-check движок** (план пола: свободно/занято → размещение товара с зазорами → да/нет + где).
3. **Масштабная вставка** товара + вердикт «влезает / велик на N см».
4. **Confidence-таксономия** (measured / estimated / partial / not_visible / need_extra_photo).
5. **Dataset 20-30 комнат** (фото+A4+рулетка) для валидации ±10% — не оптимизировать вслепую под room1.
6. **Видео-тир** (multi-view VGGT/MASt3R + A4) — полный план для ремонта/переноса.
