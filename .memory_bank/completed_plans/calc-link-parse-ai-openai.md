## Цель
Повысить точность автозаполнения по ссылке: когда детерминированный парсер не нашёл поле — дочитать
страницу через OpenAI (ChatGPT) и вернуть СТРОГО нужные значения, только реально найденные (без выдумывания/округления).

## Источник задачи
Владелец: ИИ-фолбэк на OpenAI (не Gemini); «перед предложением посмотри, как сделано сейчас».

## Как сделано СЕЙЧАС (просмотрено)
- `app/api/calc/parse-link/route.ts`: fetch страницы (лимит 2 МБ, UA, таймаут 8с) → `parseProductHtml`.
- `lib/calc/link-parse.ts`: чистая функция, детерминированные regex по og-мете + тексту без тегов, по видам
  (обои: рулон W×L + раппорт; плитка/ламинат: AxB, см→мм; цена: og/itemprop/«N ₽»). Заполняет ТОЛЬКО
  найденное — НЕ выдумывает. Клиент `LinkAutofill` применяет `spec` (после UI-захода — автоподгрузка).
- OpenAI в приложении НЕ подключён (только упоминание-заглушка в `lib/providers/*`; ключ есть у
  ads-watchdog, Python, отдельно). Провайдеры приложения = Gemini (картинки/зрение). `openai`-пакета нет.

## Предлагаемая логика (гибрид, без выдумывания)
1. Детерминированный парсер — как есть, ПЕРВЫМ (точно/бесплатно/быстро).
2. Если после него ПУСТЫ ключевые поля вида → **ИИ-фолбэк**: отдать OpenAI текст страницы (со снятыми
   тегами, обрезка ~12k симв. для токенов) + список нужных полей → вернуть JSON, только найденное,
   `null` где нет, БЕЗ догадок/округлений. Провалидировать (Zod, как `MaterialSpec`), смёржить в spec
   ТОЛЬКО в пустые поля (детерминированный результат приоритетнее).
3. Нет ключа `OPENAI_API_KEY` / ошибка/таймаут ИИ → тихо пропускаем (текущее поведение сохраняется).
4. Анти-бот-маркетплейсы (Озон/ВБ): если саму страницу не удалось скачать — ИИ не поможет (нет текста);
   там ручной ввод (позже — headless/фиды, отдельно). Google-поиск НЕ используем.

## Скоуп — что входит
- [ ] `lib/calc/link-parse-ai.ts` (нов.) — вызов OpenAI `chat/completions` через `fetch` (без SDK-зависимости),
      JSON-mode, дешёвая модель (по умолчанию `OPENAI_EXTRACT_MODEL`, напр. `gpt-4o-mini` — уточнить у владельца),
      строгий промпт «только найденное, не выдумывай, не округляй», таймаут ~10с, Zod-валидация ответа.
- [ ] `app/api/calc/parse-link/route.ts` — после детерминированного парса, если пусты ключевые поля и есть
      ключ → вызвать ИИ, смёржить в пустые; ответ формата как сейчас (`{ok,title,priceRub,spec}`).
- [ ] `lib/env.ts` + `docker-compose.yml` — `OPENAI_API_KEY` (+ опц. `OPENAI_EXTRACT_MODEL`) в env приложения.
- [ ] `.env.example` — раскомментировать/описать `OPENAI_API_KEY`.
- [ ] тесты: merge-логика (ИИ дозаполняет только пустое, не перетирает детерминированное) с моком fetch.

## Скоуп — что НЕ входит
- Per-site парсеры под конкретные магазины — отдельными планами по мере реальных ссылок.
- Headless-браузер/официальные фиды для анти-бот-маркетплейсов — позже.
- Реальный поиск «дешевле»/матчинг — не здесь.

## Критерии приёмки
- [ ] На «сложной» ссылке (где regex не всё взял) ИИ дозаполняет недостающее корректно; на простой — ИИ не зовётся.
- [ ] Без ключа/при ошибке ИИ — поведение как сейчас (детерминированный результат), без падений.
- [ ] ИИ не перетирает найденное детерминированно; пустое остаётся пустым, если ИИ не нашёл.
- [ ] `typecheck && lint && test && build` + e2e зелёные (CI без ключа → ИИ-ветка не зовётся).
