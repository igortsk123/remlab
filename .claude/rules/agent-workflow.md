# Workflow агента

Агент НИКОГДА не пишет код без явного разрешения. Полная версия с обоснованиями — `guides/agent-workflow.md`.

## Двухэтапный процесс

**Этап 1 — план:**
1. Прочитать `CLAUDE.md` + `.memory_bank/INDEX.md`.
2. Прочитать файлы по теме (decision tree в INDEX → Tier 1 → Tier 2 при нужде).
3. Составить план по `.memory_bank/plans/_template.md`.
4. Сохранить как `.memory_bank/plans/<slug>.md` со статусом `draft`.
5. Показать план в чате. Написать: «План готов. Жду команду "деплой".»

**Этап 2 — «деплой»:**
1. Прочитать план из `plans/<slug>.md`, сменить статус на `in_progress`.
2. Перечислить файлы к изменению — не трогать остальные.
3. Выполнить. Запустить lint/build/тесты — исправить ошибки.
4. Обновить Memory Bank (**ОБЯЗАТЕЛЬНО перед `completed`**, `memory-discipline.md`):
   - затронутые `core/*` сводки обновлены (`updated:`); новая крупная область → создана `core/<домен>.md`;
   - решения → `decisions.md` (ADR); смена этапа → `project-state.md` (снимок ПЕРЕПИСАТЬ,
     историю — в `changelog/project-history.md`);
   - запущен `/memory-check` (захват → уровни → связи+INDEX → чистота); audit «чисто».
5. Только после этого: `completed` + дата + completion summary → перенести в `completed_plans/`
   (реестры пересоберёт аудит). Частично → `partial`, остаётся в `plans/`.

**Гейт:** план НЕ может стать `completed`, пока `/memory-check` не выполнен и audit не «чисто».
Это единственная жёсткая точка, где ловится «поработал → не записал».

## Статусы плана
| Статус | Значение |
|--------|----------|
| `draft` | Создан, ждёт «деплой» |
| `in_progress` | Деплой начат |
| `partial` | Прерван частично — остаётся в `plans/` |
| `completed` | Готов → переносится в `completed_plans/` |
| `cancelled` | Отменён явно — остаётся в `plans/` |

**Только `completed` переносятся в `completed_plans/`.**

## Выкатка после изменений (стандарт — постоянное разрешение владельца, 2026-07-02)
Любое завершённое изменение кода ВСЕГДА доводится до прода — не оставляем в ветке/локально:
1. `pnpm typecheck && pnpm lint && pnpm test && pnpm build` — зелёные (иначе не катим, чинить).
2. Закоммитить; смерджить в `main` (ff), `git push origin main` (GitHub — CI-гейт).
3. Выкатить на сервер: `./deploy.sh` (бэкап БД + smoke + авто-откат на `:prev`; VPN-нода `remnanode`
   изолирована — не трогать). Прод не оставлять позади `main`.
- Владелец дал **постоянное разрешение на push + deploy И на правки сервера** (Caddy/compose/статик-роуты)
  — отдельно НЕ переспрашивать (решение владельца 2026-07-05, отменяет прежнее «согласовывать правки сервера»).
- Всё равно обязательно: бэкап изменяемого конфига перед правкой, обратимость (rollback), **VPN-нода
  `remnanode` изолирована — не трогать**. Предупреждать только про реально необратимое (потеря данных/миграции БД).
- Playbook и детали сервера — `.memory_bank/deployment.md`, `deploy.sh`.

## Изоляция фич
- Ветки: `feature/<area>`. Не трогать файлы чужой фичи.
- Нужен чужой компонент — импортировать, не копировать.

## Контекст и /clear
- `/clear` перед новой задачей. После 3 PR подряд — рекомендовать `/clear`.
- Перегруженный контекст до плана — попросить `/clear`.
