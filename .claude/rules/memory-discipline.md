# Дисциплина памяти

Память — процедурная, а не накопительная: обновляется по ходу работы, не «когда-нибудь».

## Два слоя памяти — и мост между ними (КРИТИЧНО)
- **`.memory_bank/` (в git, канон проекта)** — ВСЁ проектное durable: архитектура, схема, флоу,
  решения, интеграции, цифры, состояние. Каноничная `.memory_bank/` — **ОДНА на проект, в
  репозитории кода**. Две копии (docs-директория + репо) = дивергенция; если так вышло — сведи к
  одной (перенеси уникальное, вторую замени README-указателем). Audit ловит это как `DIVERGENCE`.
- **`~/.claude/projects/<sanitized-cwd>/memory/` (авто-память харнесса, per-user, вне git)** —
  служебное про работу агента: указатели на доступы, грабли инструментов, личные предпочтения.
  Харнесс пишет сюда ПО УМОЛЧАНИЮ — это НЕ хранилище проекта: вне репо, команда не видит.
- **Мост:** `/memory-check` (Этап 1.5) реконсилит авто-память → проектное переносится в банк,
  per-user остаётся. Нельзя, чтобы проектный факт жил ТОЛЬКО в авто-памяти — он невидим команде
  и потеряется (так утёк sup2: 20 файлов авто-памяти при замёрзшем банке).

## Цикл сессии
1. **Старт:** прочитать `.memory_bank/INDEX.md` → файлы по теме (decision tree).
   `project-state.md` старше ~2 недель — не доверяй снимку молча: сверь с git-логом/кодом или
   начни с `/memory-check` (SessionStart-hook напоминает об этом сам, если включён).
2. **В процессе:** узнал durable-факт → зафиксировать СРАЗУ в нужный файл `.memory_bank/`
   (no-orphan: frontmatter + аудит впишет в INDEX). Не откладывать «на потом».
3. **Конец сессии / перед `/clear` (обязательно):** запустить **`/memory-check`** — единый конвейер:
   (1) ЗАХВАТ проектного из сессии, (1.5) МОСТ авто-памяти, (2) уровни, (3) связи+INDEX, (4) чистота.
   Stop-hook напоминает, если работа была, а захват — нет. Глубокая уборка — `/memory-cleanup`.

## Когда ИЗВЛЕКАТЬ
1. Перед задачей: `INDEX.md` → `core/<тема>.md` (Tier 1) по decision tree.
2. Не хватает деталей → drill в Tier 2 (`<area>/`, `guides/`), указанный в конце сводки.
3. Не сканируй всё подряд — иди по указателям (`tier2:`, `[[ссылки]]`).
4. **Anti-rediscovery:** перед реализацией интеграции/фичи проверь, нет ли её уже — `grep` по коду
   (`integrations/`, `services/`, `lib/`) + сверка с `core/access-and-integrations.md` (реестр
   интеграций проекта; если его ещё нет — заведи). НЕ переоткрывать и не переписывать существующее.

## Когда СОХРАНЯТЬ / ОБНОВЛЯТЬ
Триггеры — изменилось что-то из:
- **архитектура** (новые слои/папки/паттерны) → `core/architecture.md` + Tier 2;
- **маршруты / флоу** → соответствующая `core/*` + Tier 2;
- **контракты / API / модели** → `domain/` + `core/data-models.md`;
- **принятое решение** → `decisions.md` (ADR: Дата/Решение/Почему/Альтернативы/Влияет на);
- **внешняя интеграция / сторонний API** (эндпоинт, ГДЕ ключ/прокси, формат запроса-ответа,
  клиент в коде) → `core/access-and-integrations.md`;
- **новая крупная функциональная область** → ОБЯЗАТЕЛЬНО `core/<домен>.md` (иначе decision tree
  не растёт, и для агента домена «не существует»); audit подсветит `NO-TIER1`;
- **крупный merge / смена этапа** → `project-state.md`: снимок ПЕРЕПИСАТЬ (+`updated:`),
  вытесненную хронологию — в `changelog/project-history.md`.

## Куда писать
| Что | Файл |
|-----|------|
| Архитектурное решение | `decisions.md` |
| Текущее состояние проекта | `project-state.md` (снимок ≤ ~10 KB; история → `changelog/project-history.md`) |
| Задача в работе | `plans/<slug>.md` |
| Доменная модель | `domain/<entity>.md` (+ парная `core/*` сводка) |
| Короткая сводка темы | `core/<тема>.md` (≤3 KB, со ссылкой `Tier 2:`) |
| Доступы/ключи (где лежат И значения) | `_secrets/ACCESS.md` — вне git; в остальных доках только указатель |

## Sync Tier 1 ↔ Tier 2 (обязательно)
- Правишь Tier 2 → обнови в нём `updated:` и **сверь парную Tier 1 сводку** (указатель `tier1:`).
- Создаёшь сводку → проставь `tier2:` вниз. Рассинхрон (`Tier2.updated > Tier1.updated`) ловит
  `memory-audit` (STALE); отставание слоя от project-state — LAGGING.

## No-orphan (обязательно)
- Новый memory-док обязан иметь frontmatter (`tier/topic/scope/updated`). Decision tree и реестры
  (core/plans/completed_plans) регенерирует аудит — руками таблицы не правим.

## Гигиена и очистка
- **Свод + гигиена** (конец сессии / после работы): `/memory-check` — единый конвейер:
  ЗАХВАТ из сессии → МОСТ авто-памяти → уровни (Tier 1/2) → связи+INDEX → структурная чистота.
  Node не нужен (`tools/memory-audit.mjs` — детерминированная структурная часть, для хука/CI).
- **Глубокая очистка** (после крупных изменений/merge/смены этапа, перед хендоффом): `/memory-cleanup` —
  дубли, устаревшее по смыслу, классификация, **архивация** (не удаление), лог. Правила: `.memory_bank/CLEANUP_POLICY.md`.
- Устаревшее не удаляем — переносим в `.memory_bank/archive/`; действия пишем в `changelog/memory-log.md`.
- Поля жизненного цикла (`status`, `source_of_truth`, `last_verified`, `review_after`) проверяются
  аудитом (REVIEW/UNVERIFIED) — заполняй у canonical и tier-1 доков, см. `METADATA_SCHEMA.md`.

## Принципы
- **Проектное durable — фиксировать ВСЕГДА** (архитектура, интеграции, решения, цифры, ход работы),
  без напоминаний. Не относящееся к проекту (личные/сторонние просьбы) — НЕ фиксировать.
- **Гейт завершения:** план не `completed`, пока `/memory-check` не выполнен и audit не «чисто»
  (`agent-workflow.md`).
- Не дублировать факт — один источник, остальные ссылаются (`[[name]]`).
- Provenance: доки из intake несут `source: _intake/...`; ручные — `source: manual`.
- Завершённые планы → `completed_plans/` (не раздувают активный контекст).
- Авто-сейв контента не делаем (нужно суждение). Напоминают и проверяют хуки (Stop/SessionStart)
  и `tools/memory-audit.mjs`; записывает — агент через `/memory-check`.
